# Signals reference

Every signal contributes `weight × intensity` points, where intensity ∈ [0, 1]
is usually a linear `ramp(value, low, high)` — signals fade in rather than
switching on. The final score is `clamp(round(sum), 0, 100)`. Weights sum to
more than 100 on purpose: no single signal reaches "high" alone, and a PR
does not need to trip everything to get there.

Levels (defaults): `low` < 20 ≤ `moderate` < 45 ≤ `elevated` < 70 ≤ `high`.

All numbers below are the defaults from `src/config.ts` and can be overridden
— see [configuration.md](./configuration.md).

## Diff signals

### `oversized-diff` (weight 18)

Total changed lines (`additions + deletions`, preferring the GitHub API's
counts since it truncates large diffs). Ramps from 400 to 4,000 lines.

*Why:* single-batch mega-diffs are the core shape of bulk-generated PRs, and
they are disproportionately expensive to review regardless of origin.
*False positive story:* legitimate large refactors and vendored updates.
That is why this is 18 points, not 70.

### `generated-code-ratio` (weight 22)

Share of **added** lines living in files that look machine-produced:
lockfiles (`package-lock.json`, `Cargo.lock`, `go.sum`, …), vendored/built
directories (`node_modules/`, `dist/`, `vendor/`, …), `*.min.js`-style files,
files with an added line longer than 500 chars, or files whose added lines
carry markers like `@generated` / "Code generated by" / "DO NOT EDIT".
Ramps from 25% to 75% of added lines; skipped entirely for diffs with fewer
than 50 added lines. Computed from the parsed diff (the API does not expose
per-file line contents).

*Why:* regenerated artifacts inflate diffs and bury the reviewable part.
*False positive story:* an honest "regenerate the lockfile" PR — which is
fine, because such a PR *should* be triaged as "skim the lockfile, review the
3 manifest lines," and that is exactly what the signal tells you.

### `missing-tests` (weight 14)

Fires when ≥ 40 added lines of non-test, non-vendored code arrive with zero
test-file changes. Intensity has a 0.2 floor and ramps to 1.0 at 800 added
code lines. Never fires if the PR touches any test path.

*Known blind spots:* repos with out-of-tree tests, doc-only conventions,
snapshot directories with unusual names. Set the weight to 0 if this does not
fit your project.

### `unrelated-files` (weight 10)

Distinct top-level directories touched, ramping from 3 to 9, considered only
when the diff contains at least 8 files.

*False positive story:* monorepos. A `monorepo` preset is on the
[roadmap](../ROADMAP.md); until then, tune `unrelatedDirsLow/High` up or
zero the weight.

## Body signals

### `boilerplate-body` (weight 18)

Counts distinct matches against ~20 phrasing patterns that are heavily
over-represented in template/LLM-generated PR text ("This PR introduces a
comprehensive…", "leveraging best practices", "seamless", "production-ready",
"In conclusion", "I hope this helps", emoji-decorated bullet scaffolds, …).
One hit scores zero by design — humans say "robust" too. Ramps from 1 to 7
distinct hits.

The pattern list lives in `src/signals/body.ts` (`BOILERPLATE_PATTERNS`) and
is exported so tests and downstream users can inspect it. It is
English-centric in v0.1; non-English patterns are a roadmap item.

### `missing-issue-reference` (weight 6)

No `#123`, issue URL, or PR URL anywhere in the title or body. Fixed
intensity. Deliberately the smallest weight: plenty of healthy projects
accept unlinked PRs.

### `missing-verification-steps` (weight 10)

For PRs with ≥ 20 added code lines: the body contains no fenced code block
and no testing/verification vocabulary ("tested", "reproduce", "npm test",
"screenshots", …).

*Honest limitation:* keyword-based, so an unsubstantiated "thoroughly tested"
passes the check. This is documented in the signal's own `details` text.

### `sparse-body` (weight 10)

Trimmed body under 30 characters. Intensity ramps with the size of the change
(50 → 600 changed lines) — an empty body on a two-line fix is fine.
When the body is sparse, `missing-issue-reference` and
`missing-verification-steps` are **skipped** (recorded in notes) so the same
underlying fact is not counted three times.

## Contributor signals

Handled with the most care, because this is where a tool like this can hurt
people. Rules:

- Being a new contributor is **never** a signal by itself.
- Bot authors (`type: "Bot"`, `login` ending `[bot]`) skip contributor
  signals entirely, with a note.
- Missing `author_association` degrades to a note, never a guess.

### `first-contribution-large-change` (weight 16)

`author_association` ∈ {`FIRST_TIME_CONTRIBUTOR`, `FIRST_TIMER`, `NONE`}
**and** total changed lines ramping from 300 to 2,500. The summary text
frames it as a sequencing question ("worth an early scope conversation"),
not a doubt about the person.

## LLM signal (opt-in)

### `llm-review` (weight 10)

Only exists when `--llm` is enabled and a key is configured. The model
returns `{"concern": 0..1, "note": "..."}`; concern becomes the intensity.
Bounded by design: the model can nudge the score, never dominate it.
See [llm.md](./llm.md).

## Worked examples

The five fixtures under `test/fixtures/` are the reference behavior,
enforced by `test/fixtures.test.js`:

| Fixture | Shape | Score (defaults) | Level |
| --- | --- | ---: | --- |
| `good-focused-fix` | 25-line linked bugfix with a regression test | 0 | low |
| `good-feature` | 343-line feature, tests + docs, verification notes | 0 | low |
| `slop-mega-dump` | 6,003 lines, 80% lockfile/bundle, template body, first-time author | 100 | high |
| `slop-boilerplate-body` | 306 lines, boilerplate body, no tests, known contributor | 40 | moderate |
| `slop-drive-by` | 2,101 lines across 9 top-level dirs, body: "Improvements." | 53 | elevated |

Note how `slop-boilerplate-body` lands at *moderate*, not high: a templated
description alone is a nudge, not a verdict. That gradient is the point.

## Tuning

If a signal keeps misfiring for your project: turn it down, or off, in your
config — and please also file a
[signal feedback issue](https://github.com/kero168/slopguard/issues/new?template=signal_feedback.yml)
so the defaults improve for everyone.
